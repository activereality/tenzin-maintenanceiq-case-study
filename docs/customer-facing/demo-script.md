# Demo Script

This script is a lightweight guide for a customer-facing Tenzin MaintenanceIQ walkthrough. Adapt the asset names, documents, and questions to the prospect's environment when possible.

## 1. Frame The Problem

Start with the common maintenance challenge: equipment knowledge is scattered across manuals, PDFs, shared drives, binders, notes, and technician experience. During downtime, teams need fast answers, but they also need to know where those answers came from.

Position Tenzin MaintenanceIQ as a knowledge assistant that helps teams find and use their own documentation faster. It does not replace technicians, engineers, OEM manuals, safety procedures, or existing CMMS/EAM systems.

## 2. Upload And Organize Documentation

Show how approved manuals, procedures, or reference documents can be organized around facilities and assets. Keep the explanation simple: documents are connected to the equipment they support so future questions can stay asset-specific.

Suggested talk track:

- Start with a small set of approved documents.
- Link documents to the relevant asset.
- Keep source documents traceable so users can review the original page later.

## 3. Ask A Plain-Language Maintenance Question

Ask a realistic question a technician might ask during troubleshooting, such as a symptom, alarm, reset procedure, inspection step, or component question.

Suggested talk track:

- Users should not need to remember exact document titles or page numbers.
- The question should stay tied to the selected asset or facility context.
- If reliable source material is not available, the system should say it does not have enough information.

## 4. Review The Cited Answer

Walk through the answer as a starting point for investigation, not as guaranteed maintenance instruction. Point out the cited sources and explain that citations are what make the response auditable.

Suggested talk track:

- The answer is grounded in retrieved source material.
- Citations help users check the underlying manual or procedure.
- Human judgment, safety procedures, and approved documentation remain the source of truth.

## 5. Open The Cited Manual Page

Open a cited source from the answer. Emphasize that the user can inspect the original document page behind the response rather than trusting a summary alone.

Suggested talk track:

- This reduces time spent hunting through PDFs and folders.
- It helps technicians confirm the answer against the original source.
- It supports review by maintenance managers, reliability, or engineering.

## 6. Capture Feedback Or Issue Context

Show how a user can mark whether a reference was helpful or capture issue context for future troubleshooting. Keep this focused on learning from real work without turning the pilot into a full CMMS replacement.

Suggested talk track:

- Feedback helps identify which references are useful.
- Issue context can preserve what happened and what evidence supported the next step.
- Work orders and formal maintenance records can remain in the existing CMMS/EAM.

## 7. Explain Account And Facility Data Boundaries

Close by explaining that pilot data is organized by account, facility, asset, document, and issue context. Normal users should only work with information they are authorized to see, and source documents should be opened through authenticated application paths rather than public file links.

## Suggested Close

Ask which assets, documents, and troubleshooting scenarios would make the most useful pilot. Confirm that success should be measured as validation goals, such as citation usefulness, technician usability, document lookup time, and comfort with data boundaries.
